## 2. 工作方式

每次执行任务前：

1. 先阅读 `README.md` 和 `AGENTS.md`。
2. 判断目标是否清楚。
3. 判断是否存在更简单方案。
4. 给出最小实施方案。
5. 列出计划修改文件。
6. 等待用户确认，除非用户明确要求直接执行。

每次执行后：

1. 输出修改文件清单。
2. 说明每个文件为什么修改。
3. 给出运行命令。
4. 给出验证结果。
5. 说明未解决问题。
6. 给出下一步建议。

## 3. 目录读取优先级

优先阅读：

- `README.md`
- `AGENTS.md`
- `_docs/`
- `task_queue.md`
- `CHANGELOG.md`
- `src/` 或主代码目录
- `tests/`

不要优先读取：

- `.venv/`
- `node_modules/`
- `dist/`
- `build/`
- `.git/`
- 大型缓存、导出文件、日志文件，除非任务需要。

## 4. 开发约束

- 先做最小可验证版本，避免过度工程。
- 不要修改与本任务无关的文件。
- 新增依赖必须说明原因。
- 不要硬编码密钥、路径、客户数据。
- 不要删除历史数据、测试样例、文档，除非用户明确要求。
- Windows 路径和 PowerShell 命令必须可用。
- 修改 CLI 或脚本时，必须保证 `--help` 或 README 有说明。
- 修改数据处理逻辑时，必须提供示例输入和示例输出。

## 5. 验收标准

项目当前阶段的完成标准：

- 能被 Codex 正确识别项目目标。
- 有 README、AGENTS、任务队列、变更记录。
- 有最小运行命令。
- 有最小验证命令。
- 没有硬编码密钥。
- 没有无解释的大规模重构。
